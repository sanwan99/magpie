# Contributing to Magpie

Thanks for considering contributing. Magpie is opinionated about scope —
read the principles below before opening a PR for a new feature.

## Project principles

1. **Local-first, privacy-first.** No network. No telemetry. No accounts.
   Anything that needs to phone home is out of scope.
2. **Keyboard is a first-class citizen.** Every mouse-driven action has
   a hotkey. New UI must come with a shortcut.
3. **Type-aware previews.** Cards distinguish text/code/url/image/file/folder
   visually, not by reading metadata twice.
4. **One panel.** No floating sub-windows during normal use (Settings /
   Snippet editor are separate windows by intent — they're modal-ish, not
   companion).
5. **macOS only.** No cross-platform layer. No Tauri / Electron.

If your contribution conflicts with these principles, open a discussion
before writing code.

## Development setup

Requirements:
- macOS 14 (Sonoma) or later
- Xcode 15+ (full Xcode, Command Line Tools is not enough)
- [XcodeGen](https://github.com/yonaskolb/XcodeGen) (only if you want to
  modify the project structure; the `.xcodeproj` is committed so a plain
  clone works without it)

```bash
git clone https://github.com/sanwan99/magpie.git
cd magpie
open Magpie.xcodeproj
# Build & Run (⌘R) in Xcode
```

If you change `project.yml`, regenerate:
```bash
xcodegen generate
```

## Running tests

```bash
xcodebuild test \
  -project Magpie.xcodeproj \
  -scheme Magpie \
  -destination 'platform=macOS,arch=arm64' \
  -derivedDataPath build
```

All tests run against in-memory databases and don't touch your real
`~/Library/Application Support/Magpie/clips.sqlite`. Adding a new
data-touching test? Use `Magpie.Database.makeInMemoryForTests()`.

## Style

- **Swift API Design Guidelines** — boring, readable, descriptive names.
- Comments explain *why*, not *what*. The why often refers back to a spec
  section in `prototype/剪切板工具/Magpie 原型说明.html` or a previous
  changelog (`md/codex/plan/changelog/`).
- One actor model: most code is `@MainActor`. Background work goes through
  `Task` with explicit isolation.
- Tests: XCTest, descriptive method names (`testKeepDaysDoesNotDeletePinned`),
  setUp/tearDown to keep tests independent.

## Adding a new theme (flavor)

See `Sources/Magpie/Theme/README.md`. Three small edits:

1. Add an enum case to `Flavor`
2. Add a `displayName`
3. Add a `FlavorTokens` entry

`CaseIterable` picks it up everywhere automatically.

## Submitting a PR

1. **Discuss first** if your change is non-trivial (anything beyond a bug
   fix or polish). Open an issue with the proposal.
2. **One feature per PR.** Don't bundle unrelated changes.
3. **Add tests** if you touch testable code (parsers, repositories, regex,
   timers). UI-only PRs can ship without tests but should include before/after
   screenshots.
4. **Update changelog** in your PR description. Don't write to
   `md/codex/plan/changelog/` directly — those are owned by the maintainer.
5. **Run the test suite** locally before submitting.

## Things we won't merge

- Network functionality (telemetry, sync, auto-share)
- AI summarization / suggestions
- Cross-platform layers (Win/Linux ports — fork it instead)
- Things that violate the principles at the top

## Areas that genuinely need help (if you're looking)

- Better CJK input method handling for `;sig` auto-expand
- More secret-detection regexes (catch more credential patterns without
  false positives)
- App icon designs (we'd love a real shipping icon)
- Localization (English / 简体中文 first; others welcome)
- Performance profiling on large clip histories (10k+ items)
